# ADR-007 — 用显式上下文隔离规划分支

- 状态：提议
- 日期：2026-07-18
- 规范语言：英文
- 英文规范文本：[`ADR-007-explicit-planning-contexts.md`](ADR-007-explicit-planning-contexts.md)
- 相关 Profile：[`HWM 装修规划 Profile v0.1`](../../spec/profiles/renovation-planning/v0.1/README.zh-CN.md)

## 背景

认知依据区分 Claim 是怎样被认识的，却不能标识 Claim 属于哪个规划世界。单方案夹具可以要求 Design Option 标识，却不能证明两套不兼容装修方案在模拟、比较、推荐与家庭选择过程中始终隔离。

## 决定

1. 装修规划 Claim 使用包含 `context_id`、`context_kind` 与 `revision` 的显式 `designContext`。
2. 本 Profile 定义 `base_design`、`design_option` 与 `comparison` 三类上下文。它们是 Profile 结构，不是 HWM Core 实体类。
3. 每个方案通过 `prov:wasDerivedFrom` 显式连接共享基础设计。
4. 分支内 Claim 只能消费同一方案、声明的基础设计，或需求和目录声明等无上下文共享输入。
5. 跨分支推导只能发生在列出全部 `input_context_ids` 的 `comparison` 上下文中。
6. 每个方案先独立计算需求状态，再进行跨分支比较。
7. 推荐不是 Authorization Decision；家庭选择是规划 Attestation，不是购买、安装、调试或动作派发。
8. 选择不会删除被拒方案；后续变化通过追加 Claim 或 revision Record 表达。
9. 目标语义投影把基础与方案上下文标为 `prov:Plan`，把比较标为 `prov:Activity`，并使用 `prov:used`、`generated` 与 `wasDerivedFrom` 表达沿袭。这些外部类型不能替代 Profile 的分支闭合规则。
10. 各方案需求状态在目标绑定中使用 EARL Assertion／TestResult outcome；家庭选择使用 Schema.org `ChooseAction`；`requirementStatus` 与 `selectedOption` 只作为兼容投影。
11. Schema.org `Recommendation` 可以表达建议结果，但在 Schema.org 仍把它标为新术语时不能成为唯一规范绑定。禁止从推荐自动升级为选择、购买、授权、安装或派发，仍是 HWM Profile 不变量。
12. 每个 Schema.org `ChooseAction` 都作为不可变历史事件保留。后续选择状态 Claim 使用 `dcterms:replaces`，不改写早先状态。当前选择在 `as_of` 时刻从未被替代的头部解析；分叉、成环与目标不可用分别保留为冲突或不确定。
13. 比较是封闭输入操作。只有每个已声明输入恰有一个可用、已接受、当前、时间上生效或无边界、且适用的解析结果，并且不存在未声明输入时，候选推荐才能被接纳。其他情况一律产生零条推荐。
14. Profile 管推荐准入，不管排序；提出具体方案的算法不属于 HWM。
15. 每个 Design Option 恰有一个显式基础设计派生。链接缺失或存在歧义属于结构无效；依赖已知但不可取得时结果不确定；完整但根植于非当前基础的分支只能作为历史。
16. 保留不可变派生 Claim 与把它接纳为当前结果是两件事。依赖被删除、数据源故障或无权访问都不会改写派生历史，但会阻止当前模拟、求值和推荐。
17. 基础和方案保留大量原内容的修订使用 `prov:wasRevisionOf`；`prov:invalidatedAtTime` 可以标记旧 Plan 何时不再可供后续推导。任一关系都不会执行隐式 rebase。
18. Rebase 会创建方案新 revision、连接新的显式基础派生并重新生成下游结果。绑定旧方案 revision 的输出继续作为历史，不能带入新分支。
19. 同一次当前比较中的每个分支都闭合到同一个当前基础根。不同布局备选必须共享显式更高层基础，不能绕过依赖闭包。

## 后果

- Agent 可以比较反事实家庭设计，而不污染当前或未来物理世界。
- Resolver 必须理解上下文，并在缺少必要上下文或派生关系时关闭失败。
- 比较沿袭可检查，被拒方案仍可审计或重新考虑。
- 家庭可以从推荐方案改选其他方案，而不改写推荐或早先选择历史。
- 不完整证据不能静默变成建议；格式错误与知识不可用都关闭失败，但结果状态保持区分。
- 基础被删除或隐藏不会让旧模拟自动变成错误，但会使其当前用途无法证明。
- 更新户型图不能静默保留基于旧 revision 计算的照明、电气或需求求值结果。
- 模型增加 Profile 约束与临时规划谓词，但不增加 Core 原语。
- 两个临时谓词从目标语义词汇中移除；来源成熟度被显式保留，而不是隐藏在名义复用之后。

## 当前证据

装修规划知识包包含一个基础设计、两个互斥照明方案、各自模拟与需求求值、一次显式比较和一次业主选择。独立分支解析 oracle 现在包含 31 个用例：5 个选择、11 个推荐准入、10 个基础依赖和 5 个比较沿袭用例。JavaScript 与独立 Python reader 能区分沿袭格式错误、知识不可用和仅历史分支；证明基础被删除或无权访问时不产生当前输出；要求显式 rebase 与结果重算；拒绝混合基础的当前比较；同时保留所有旧输出。两条路径都不会产生 `BuyAction`、Authorization、安装或派发。
